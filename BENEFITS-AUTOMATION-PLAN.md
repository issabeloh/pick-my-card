# AI 自動化權益監控與更新系統 — 實施規劃書

> 目標：把「巡邏銀行官網 → 貼給 GEM → 手動貼回 Google Sheet」的全手動流程，
> 變成「系統自動發現變化 → AI 自動解析 → 你只按一顆核准鍵」的半自動流程。
>
> 設計原則：**零月費、不架伺服器、全部住在你已經在用的 Google Sheets + Apps Script 裡。**

三條鐵則（整份規劃的靈魂）：

1. **AI 只做閱讀理解，程式做格式與數學。** AI 輸出被鎖死欄位的 JSON；
   Tab 分隔、cap = 上限÷率、組別編號……全部交給寫死的程式。這是精準度問題的根治法。
2. **AI 永遠不直接碰正式資料。** 一切先進「待審核」工作表，你核准了程式才寫入。
3. **比對新舊是數學題，不是 AI 的閱讀題。** 變多/變少/不變由程式比數字，100% 可靠。

## 1. 先診斷：為什麼現在的 GEM 流程不準、又累

| 問題 | 原因 | 解法 |
|---|---|---|
| 活動一複雜就不準 | 你要求 GEM「一次做兩件事」：理解活動 **＋** 排出 26 欄以上的單列 TSV。排版佔掉模型大量注意力，複雜活動時兩頭皆失 | AI 只輸出 JSON（見 §3.1），排版交給程式 |
| 欄位有時多、有時少、有時錯位 | 聊天視窗（GEM）**無法強制輸出格式**，模型「盡量配合」而已 | 改走 API 並開啟「結構化輸出」（JSON Schema），欄位名與型別被鎖死，不可能多欄少欄 |
| cap 公式、promo_id 偶爾算錯 | 你讓 AI 做算術和編流水號——這兩件事 LLM 天生不擅長 | 程式算。AI 只回報「上限 200 元、回饋率 7%」兩個原始數字 |
| 流程高度手動 | GEM 只能人貼進去、人貼出來 | Apps Script 直接呼叫 API，貼上這個動作徹底消失 |
| 不知道官網何時更新 | 沒有監控機制，靠人巡 | 第一階段解決（§2） |

> 好消息：你四個 GEM 的 instruction 內容（欄位定義、命名規則、排除清單邏輯）
> **90% 可以直接沿用**，只是搬家：從「GEM 的說明欄」搬到「API 呼叫的 system prompt + JSON Schema」。
> 你累積的規則一點都不浪費。

## 2. 第一階段：前置監控與情報收集

### 2.1 觸發機制 — 用你已經有的 Apps Script（$0）

Google Apps Script 內建「**時間驅動觸發器**」：設定好之後，Google 每天在指定時段自動幫你跑一次腳本，
不用開電腦、不用伺服器、完全免費。這就是整個系統的鬧鐘。

建議排程：每天早上 07:00 跑一輪（銀行公告多在上班日發佈；一天一次對這個場景完全夠用）。

### 2.2 監控清單 — 新增一個 `Watchlist` 工作表

你用「填表」的方式管理要監控什麼，之後新增卡片只要加一列，不用改程式：

| 欄位 | 說明 | 範例 |
|---|---|---|
| card_id | 對應 Cards Data 的 id | `yushan-unicard` |
| bank | 銀行名（純備註） | 玉山銀行 |
| url | 要監控的網址 | 卡片權益頁網址 |
| watch_type | `權益頁` / `公告頁` / `新戶活動頁` | 權益頁 |
| css_selector | 選填：只看網頁的哪一塊（不會填就留空，看整頁正文） | `#main-content` |
| last_snapshot | 程式自動維護：上次抓到的正文 | （自動） |
| last_checked | 程式自動維護：上次檢查時間 | （自動） |
| active | TRUE/FALSE | TRUE |

**每張卡建議放 1~3 個網址**，其中「銀行公告/最新消息頁」比行銷首頁重要——
權益調整依法要公告，公告頁是最早、最準的訊號源，而且通常是簡單網頁（好抓）。

### 2.3 如何精準抓取、不被雜訊干擾（三道濾網）

銀行官網每天都有小變動（輪播圖、日期、廣告 banner），直接比對整頁 HTML 會天天誤報。三道濾網：

1. **只比正文**：抓回網頁後先去掉程式碼、選單、頁尾，把連續空白壓成一個，只留下人看得到的文字。
   存這份「乾淨文字」當快照。（若該列有填 css_selector，就只看那一塊，更乾淨。）
2. **關鍵字閘門**：新舊文字有差異時，取出「變動的那些段落」，必須含至少一個關鍵字才算事件：
   `回饋、加碼、%、權益、活動、調整、終止、停止、新戶、上限、登錄`。
   輪播圖文案、頁尾版權年份的變化過不了這關。
3. **差異量門檻**：變動少於約 30 個字直接忽略（日期自動更新之類的假警報）。

**重要細節**：快照要存「全文」而不是只存指紋（hash）——
因為第二階段要把「舊版 vs 新版的差異」餵給 AI，只有指紋就沒有差異內容可餵了。
全文存在 Watchlist 同列的儲存格即可（一格可放 5 萬字，夠用；超長頁面可存到 Drive 文字檔）。

### 2.4 抓不到怎麼辦（少數難搞的銀行）

有些銀行網頁是「先載入空殼、再用程式填內容」（JS 動態網頁），Apps Script 抓回來會是空的；
極少數會擋機器人。對策依序：

1. **換監控目標**：該銀行改監控「公告列表頁」——公告頁幾乎都是傳統靜態網頁，抓得到。
2. **免費外部工具補位**（只給難搞的那幾家用）：
   - [changedetection.io](https://changedetection.io)：開源免費（自架），有支援動態網頁的模式
   - Distill Web Monitor：瀏覽器擴充功能，免費版可在你自己的電腦監控 25 個頁面
   - 這些工具的角色只是「**叫你去看**」（發通知），看到後把文字貼進情報收件匣，後面流程照走
3. **工程備援（先不做）**：GitHub Actions 免費額度 + 無頭瀏覽器可以全自動抓動態網頁，
   但這超出非工程師維護範圍，等前面都跑順、且真有多家銀行抓不到再說。

### 2.5 第一階段的產出 — 「情報收件匣」工作表

| 欄位 | 內容 |
|---|---|
| 日期時間 | 偵測到變化的時間 |
| card_id / 銀行 / 網址 | 哪張卡、哪一頁 |
| 變化摘要 | 變動段落的前 200 字 |
| 舊文字 / 新文字 | 完整差異內容（給第二階段的 AI 用） |
| 狀態 | `待解析` → `已解析` → `已審核` |

同時寄一封 Email 給你（Apps Script 內建寄信功能，零設定）。
到這裡為止，你就從「不定期手動巡 20 幾個官網」升級成「**有事才收到信**」。

## 3. 第二階段：AI 解析、比對、寫入

### 3.1 從 GEM 搬到 API — 精準度躍升的關鍵一步

Apps Script 可以直接呼叫 Gemini API（`UrlFetchApp` 一個 POST 就好；免費額度對這個用量綽綽有餘，
每天最多幾十次呼叫）。搬家後拿到三個 GEM 給不了的能力：

1. **結構化輸出（responseSchema）**：把欄位定義成 JSON Schema 附在 API 呼叫裡，
   模型的輸出被強制符合格式——欄位名、型別（數字/文字/布林）、必填與否全部鎖死。
   「有時多一欄、有時錯位」這類問題從根源消失。
2. **一次只解析一個活動**：先用一次便宜的呼叫把整頁「切成活動清單」，再逐一解析。
   你觀察到「活動變複雜就不準」，正解就是切小——模型一次只需要看懂一個活動。
3. **每個數字都要附原文引用（evidence 欄位）**：schema 裡要求 AI 對每個回饋率/上限
   附上「我是從這句原文讀到的」。好處有二：你審核時一眼能對答案；AI 被迫誠實，幻覺大幅下降。

另外兩個 prompt 設計調整：

- **「不確定就停下來問我」→ 改成 `needs_review` 旗標**：自動化流程裡 AI 不能停下來聊天。
  改成 schema 裡放 `needs_review: true/false` + `question: "官網未列排除清單"`，
  這些項目在待審核表會被標黃，你一眼看到 AI 哪裡沒把握。
- **AI 不做算術、不編號**：cap 公式（`=200/0.07`）、promo_id 流水號、Excel 剩餘天數公式，
  全部改由程式生成。AI 只回報原始數字（上限金額、回饋率、日期）。

### 3.2 你現有四個 GEM 的遷移對照

| 現有 GEM | 遷移後 | 重點變化 |
|---|---|---|
| 卡片權益（26 欄 TSV） | `parseCardBenefits()` | rate 組別變成 JSON 陣列（每個活動一個物件：率/上限金額/通路/條件/期間）；「14 組別」「hide 組別」的編號規則由**程式**套用，AI 只標記「這是廣告類」「這是國外消費」 |
| 新戶活動 | `parseNewPromo()` | promo_id 由程式用日期自動編；bonus_cap 公式由程式生成；promo_types 用 schema 的選項清單（enum）鎖死只能填那三種 |
| 領券型活動 | `parseCoupon()` | 兩個 Excel 公式欄（剩餘天數/活動期間）100% 程式生成——這正是現在最容易錯的「欄位代碼計算」，以後根本不經過 AI |
| tags 分類 | 查表優先，AI 補位 | 建一個「通路→分類」對照表，程式先查表（免費、瞬間、永遠一致）；查不到的新通路才問 AI，AI 的答案存回對照表 → **越用越準、越用越省** |

### 3.3 自動比對新舊資料（變多？變少？不變？）— 交給程式，不是 AI

比對是數學題。AI 負責把官網文字變成數字，數字比大小由程式做，100% 可靠：

```
AI 解析出新 JSON ──┐
                   ├──→ 程式逐項比對 ──→ 變更報告
Cards Data 現有值 ──┘

比對邏輯（寫死在程式裡）：
  1. 用 card_id 撈出這張卡目前的 rate_N / cap_N / items_N …
  2. 用「通路組」對齊：新資料的 items 跟舊資料哪一組重疊最多，就視為同一組
  3. 對齊後比數字：
       新 rate > 舊 rate  → ▲ 變多
       新 rate < 舊 rate  → ▼ 變少（最需要通知的！）
       相同               → ═ 不變（連通知都不用發）
       舊有新無           → － 疑似下架（標紅，一定人工看）
       新有舊無           → ＋ 新增活動
  4. 通路組完全對不上（銀行改名、拆併活動）→ 這一種情況才請 AI 幫忙配對，
     且 AI 的配對結果一樣進待審核，不直接採用
```

### 3.4 安全寫入 — 絕不弄亂原本的資料結構

**鐵則：AI 和自動程式都不直接寫 Cards Data 正式表。** 中間隔一層「待審核」工作表：

| 核准☐ | 卡片 | 項目 | 欄位 | 舊值 | 新值 | 方向 | AI信心 | 原文引用 | 備註 |
|---|---|---|---|---|---|---|---|---|---|
| ☐ | 玉山Uni | 指定通路 | rate_3 | 3 | 2 | ▼變少 | 高 | 「…調整為2%…」 | |
| ☐ | 富邦J | 日韓消費 | cap_2 | 66667 | 100000 | ▲變多 | 高 | 「…上限提高至…」 | |
| ☐ | 台新Richart | （新活動） | rate_9(新) | — | 3.8 | ＋新增 | 中 | 「…」 | needs_review：未列排除清單 |

- 一列＝一個變更，粒度細到單一儲存格
- 你勾「核准」→ 按選單裡的「套用核准變更」→ 程式**只改被核准的那些格子**，
  同卡片其他 16 組 rate、levelSettings JSON、任何你手動維護的欄位完全不動
- **寫入前自動驗證**（沿用你 runQACheck 的精神）：rate 必須是數字、日期格式必須合法、
  cap 公式必須算得出來、card_id 必須存在——任何一項不過就退回待審核並標紅，寧可不寫也不寫壞
- 寫入後照你**現有流程**：runQACheck → exportToJSON → 更新 cards.data ＋ cards.version，一步都不改
- 保險絲：Google Sheets 內建「版本紀錄」（檔案→版本記錄），任何寫壞都能一鍵還原到寫入前——
  這是免費的復原鍵

### 3.5 通知你

- **Email（推薦起手式）**：Apps Script 內建 `MailApp`，零設定零成本。兩種信：
  1. 「偵測到官網變動」——含變動摘要與連結（第一階段發）
  2. 「待審核有 N 筆變更：▲2 ▼1 ＋1」——含變更表，紅綠標色（第二階段發）
- **想要手機即時推播**：Telegram Bot 免費、Apps Script 一個 POST 就能發。
  ⚠️ 注意：**LINE Notify 已於 2025/3/31 終止服務**，別規劃那條路；
  LINE 官方替代方案（Messaging API）免費額度較緊且設定繁瑣，Telegram 簡單得多。

## 5. MVP：先做這 20%，拿到 80% 的效果

依價值排序。**MVP = 第 1＋2 項**，其餘之後再加：

### ✅ 第 1 優先：監控＋通知（第一階段整套）
- Watchlist 工作表＋每日觸發＋正文快照比對＋三道濾網＋Email
- **為什麼價值最大**：你現在最大的痛不是解析不準，而是「不知道何時更新，得自己巡」。
  這一步把巡邏時間降到零，而且永遠不漏接。
- 工作量：一個 Apps Script 檔案、約 150 行——可以直接請 AI 把整段程式寫好給你貼上，
  你要做的是「驗收」：挑 10 個網址試跑一週，看誤報/漏報，調關鍵字清單。

### ✅ 第 2 優先：API 解析 → 待審核表（先做「新戶活動」這一種）
- 挑新戶活動先行，因為它是四個 GEM 裡**結構最單純、更新最頻繁**的——最快見效、最好驗證
- 把新戶 GEM 的 instruction 搬成 system prompt + JSON Schema，Apps Script 呼叫 API，
  結果寫進待審核表（此階段先人工從待審核表複製到正式表也行）
- 這一步打通「AI→JSON→Sheet」的水管；之後權益、領券只是換 schema 複製貼上

### 之後再做（非 MVP）：
3. 比對引擎＋「套用核准變更」按鈕（§3.3、§3.4）
4. 卡片權益、領券 GEM 遷移；tags 查表化
5. 難搞網頁的外部工具備援、每週「零變動也回報」的心跳摘要

**為什麼 1＋2 就是 80%**：現在整條流程裡最花你時間的是「巡官網」和「貼來貼去」，
1 消滅前者、2 消滅後者的大半；比對和一鍵寫入很爽但屬於錦上添花——
先讓待審核表當你的工作清單，人工核對著改，也已經比現況輕鬆一個量級。

## 6. 實作路線圖（4 週，每週幾小時）

| 週 | 做什麼 | 完成的判斷標準 |
|---|---|---|
| 1 | 建 Watchlist（先放 10 個網址）＋監控腳本＋每日觸發＋Email | 手動改一個測試頁面，隔天收到通知信 |
| 2 | 調濾網（關鍵字/門檻）、擴到全部卡片；新戶活動 JSON Schema＋API 解析 | 貼一個真實新戶活動頁文字，待審核表出現正確的一列 |
| 3 | 待審核表定稿＋每日摘要信 | 一週內誤報 ≤ 2 次、漏報 0 次 |

### 7.1 新戶活動 JSON Schema 範例（節錄）

```json
{
  "type": "object",
  "properties": {
    "card_id": { "type": "string", "enum": ["yushan-unicard", "yushan-ubear", "cathay-cube", "..."] },
    "promo_types": {
      "type": "array",
      "items": { "type": "string", "enum": ["贈品", "回饋加碼", "定額點數"] }
    },
    "new_customer_definition": { "type": "string" },
    "period_start": { "type": "string", "description": "YYYY/M/D，官網寫即日起則留空字串" },
    "period_end":   { "type": "string" },
    "bonus_rate_percent": { "type": "number", "description": "只填數字，如 5" },
    "bonus_cap_amount":   { "type": "number", "description": "回饋金額上限的原始數字，如 200。公式由程式算" },
    "bonus_merchants": { "type": "array", "items": { "type": "string" } },
    "evidence": { "type": "string", "description": "支撐 rate 與 cap 的官網原文引用" },
    "needs_review": { "type": "boolean" },
    "review_question": { "type": "string" }
  },
  "required": ["card_id", "promo_types", "needs_review"]
}
```

要點：`enum` 把卡片 ID 和活動類型鎖成「只能從清單選」；cap 收原始數字，
`=200/0.07` 這種公式由程式生成；`evidence` 強制附原文；`needs_review` 取代「停下來問我」。

### 7.2 Apps Script 呼叫 Gemini API 骨架（節錄）

```javascript
function parseNewPromo(rawText, cardIdHint) {
  const apiKey = PropertiesService.getScriptProperties().getProperty('GEMINI_API_KEY');
  // ⚠️ API 金鑰放「專案設定 → 指令碼屬性」，絕不要直接寫在程式碼裡

  const payload = {
    contents: [{ parts: [{ text: '以下是信用卡新戶活動原文（卡片提示：' + cardIdHint + '）\n\n' + rawText }] }],
    systemInstruction: { parts: [{ text: NEW_PROMO_SYSTEM_PROMPT }] },  // ← 你現有 GEM instruction 搬過來
    generationConfig: {
      responseMimeType: 'application/json',
      responseSchema: NEW_PROMO_SCHEMA   // ← 7.1 的 schema
    }
  };

  const res = UrlFetchApp.fetch(
    'https://generativelanguage.googleapis.com/v1beta/models/gemini-2.0-flash:generateContent?key=' + apiKey,
    { method: 'post', contentType: 'application/json', payload: JSON.stringify(payload) }
  );
  return JSON.parse(JSON.parse(res.getContentText()).candidates[0].content.parts[0].text);
}
```

拿到 JSON 之後，由另一個純程式函數負責：生成 promo_id、算 cap 公式、
組成一列寫進「待審核」工作表。AI 的工作到 JSON 為止。
