# 判斷力 Rubrics（拿不準時查這裡，不要憑感覺）

> 每條 rubric 都有：判準（可核對的訊號）、正例（該做）、反例（不該做）。
> 正反例都以本專案為場景。判準之間衝突時，順位：用戶明示 > 本檔 > 省事。

## 1. 何時該升級模型

**判準**（符合任一即升級，升級規則見 `docs/ops/dispatch.md` 第 5 節）：
- 同一驗收條件連續 2 次沒過，且第二次的修法是在第一次修法上打補丁
- 任務需要同時推理 ≥3 個檔案的互動（如：改 Apps Script 匯出格式 → 前端解析 → 顯示邏輯的連鎖）
- 開始需要「取捨」而不是「執行」：兩個規則都對但互相擠壓，要決定犧牲哪個

**正例**：haiku scout 回報 `parseCashbackRate` 在 script.js:800，實際 Grep 是 1,100 一帶——定位錯 1 次，下次同類搜尋直接派 sonnet。
**反例**：builder 跑 preflight 失敗，原因是自己忘了跑 update-version.sh——這是流程沒走完，不是能力不足；補跑即可，**不升級**。升級是用來對付「能力邊界」，不是用來逃避「沒照 checklist」。

## 2. 何時算真的完成

**判準**（全部成立才算完成，缺一是「進行中」）：
1. 驗收條件逐條 ✅ 且每條有證據（指令輸出、read-back 內容、頁面行為）
2. `bash tools/preflight.sh` 通過，輸出已附在回報
3. 改了計算/搜尋/顯示 → `docs/ops/regression.md` 相關條目已比對
4. 已 commit + push（短命容器：沒 push ＝ 沒發生）
5. 回報已寫成用戶看得懂的結論（不是過程流水帳）

**正例**：改了 displayParkingBenefits → 跑 preflight ✅ → 手動搜「所有停車」「家樂福」比對結果 → commit+push → 回報「改了什麼＋兩組搜尋結果一致」。
**反例**：「我已經修改了 script.js 的相關邏輯，應該可以正常運作」——沒有任何一條判準成立。「應該可以」永遠不是完成。

## 3. 何時該停下來問用戶

**判準**（符合任一就問，用 AskUserQuestion，附足夠 context 讓用戶不用回捲。subagent 沒有問用戶的管道：把問題＋選項回報給主對話，由主對話代問，不要自己拍板）：
- **不可逆且動到別人的東西**：刪/改用戶儲存的資料（級別、筆記、額度）、改級別名稱、改 firestore.rules 語義、force push 共用分支
- **兩個合理解會產生用戶可見的不同行為**：例如「重複的活動要合併顯示還是逐筆顯示」——這在本專案歷史上真的來回改過，是產品決策不是工程決策
- **需求與鐵則相撞**：用戶要求的做法會違反 CLAUDE.md 鐵則（如「找不到級別就存回預設值」）——指出衝突讓用戶裁決，不要默默照做也不要默默拒絕
- 同一件事重試 2 輪仍卡住（dispatch.md 第 5 節）

**正例**：任務說「清理舊的級別資料」——「清理」可能指記憶體過濾（安全）也可能指刪 Firestore 欄位（不可逆）→ 問。
**反例**：函數命名、變數命名、註解措辭、log 文案——選慣例上最一致的做法直接做，**不問**。把這種事拿去問是把決策成本轉嫁給用戶。

## 4. 什麼訊號代表方向錯了（該換路，而不是再試一次）

**判準**（任一出現就停手換路）：
- **震盪**：修 A 壞了 B，修 B 又壞回 A
- **特例堆疊**：為了讓某個 case 通過而加的 `if` 已經是第 3 個——正確的修法通常讓特例變少，不是變多
- **診斷全靠「可能／應該／大概」**：沒有一條證據（log、diff、實跑輸出）支持目前的假設
- **改動面積遠大於症狀**：修一個顯示 NaN 的 bug 卻動了 5 個檔案
- **開始想繞過驗證**：想跳過 preflight、想把回歸清單「先註解掉」——這是方向錯誤的最強訊號

**換路的標準動作**：(1) `git checkout` 回到最後一個綠色狀態；(2) 重讀路由表指到的領域檔（十之八九有現成答案，本專案的坑幾乎都有文件）；(3) 改用「先寫出最小重現步驟」的定位法；(4) 還不行 → 帶完整軌跡升級或問用戶。
**正例**：搜「meta廣告」零結果，連改 3 處前端匹配邏輯都沒用 → 停，重讀 data-pipeline.md → 發現是匯出 guard 把 rate=0 槽丟了——問題根本不在前端（真實案例）。
**反例**：「再多加一個關鍵字比對分支試試看」——第 4 次微調同一個函數而不回頭質疑假設。

## 5. 品質底線怎麼驗（最低標準，不可再減）

任何改動收工前：
1. `bash tools/preflight.sh` 通過
2. `git diff` 自己過一遍：有沒有 debug 殘留（console.log、測試用寫死值）、有沒有無關檔案被誤改
3. 計算/搜尋/顯示改動 → regression.md 相關條目；UI 改動 → 實際開頁面看（`python3 -m http.server`），不是只看程式碼
4. 文件改動 → 交叉引用的路徑用 `ls` 驗證存在
5. 依 dispatch.md 第 6 節決定要不要派 verifier（預設：多於單點修改就派）

**正例**：改了 spotlight 輪播 → 開頁面確認換頁不跳動、hover 暫停仍有效。
**反例**：「CSS 改動很小，不用開頁面看」——本專案的視覺定案（如貼緣暗縫）就是多輪迭代的結果，小 CSS 改動踩壞既有定案的機率不低。

## 6. 模糊題與品味題（誠實條款：這套制度補不了的部分）

**判準——什麼是品味題**：正確答案不只一個、且分不出對錯只分得出「好不好看／順不順」：視覺設計取捨、文案語氣、UX 微互動、產品方向。

**弱模型（含 Sonnet 等級）遇到品味題的標準動作**：
1. **不要假裝有把握**。產出 2–3 個方案＋各自的 trade-off，讓用戶選——不要自己拍板然後被動等用戶發現不對
2. 有歷史可循的先查 `docs/project/ui-display.md`「視覺定案」與 `history.md`——很多品味題其實已經被用戶裁決過（如藍膠囊被否決、皮革框被否決），照既有裁決做，**不要重開已裁決的題**
3. 拆解、驗證、多樣本評審能提升「執行品質」，但**選哪個方向**的品味不會因為多派幾個 agent 就變準。多樣本評審適用於「有客觀優劣」的題（見 dispatch.md 第 6 節），不適用於純偏好題
4. 真的必須當場決定（用戶不在）→ 選**最保守、最可逆**的方案，並在回報裡明確標注「此處是品味判斷，我的信心低，方案 B/C 在此」

**正例**：用戶說「精選活動區看起來怪怪的」——這是品味題＋模糊題：先問哪裡怪（截圖/具體元素），不要自行大改視覺。
**反例**：把「回饋率顯示成 NaN」當品味題處理——這是有明確對錯的 bug，直接修，不用給方案。
