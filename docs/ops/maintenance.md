# 制度檔維護協議（怎麼安全地更新 CLAUDE.md 與 docs/）

> 適用檔案：`CLAUDE.md`、`docs/ops/*`、`docs/project/*`、`.claude/agents/*`、`tools/preflight.sh`。
> 原則：**制度檔是用戶的資產，不是 session 的草稿**。寧可少改，不可改壞。

## 1. 權限分級

### 🟢 可自行改（不用問，改完 commit+push）
- 在任何檔案末尾「教訓記錄」區**追加**條目（格式見第 2 節）
- 修正**確定錯誤**的事實：失效的檔案路徑、漂移的行號、改名後的函數名——「確定」= 你用 ls/Grep 驗證過現狀
- 更新環境快照（dispatch.md 開頭那段）：模型清單、agent 清單變動時，**以當下工具實際回傳為準**，並改快照日期
- CLAUDE.md 路由表**新增一列**（新建了領域檔時）
- history.md 頂部追加新的技術決策條目
- regression.md 填入/更新基準快照

### 🟡 改前需派 verifier 覆核（fresh-context 確認「改了之後規則清單無遺失、無新矛盾」）
- 改寫/精簡任何檔案的既有段落（包括第 4 節的精簡儀式）
- 修改派工模板的欄位結構（措辭微調屬 🟢）
- 修改 `.claude/agents/` 定義（model/effort/tools 調整）
- 修改 `tools/preflight.sh`（改完必須重做正反向測試：乾淨樹要 ✅、故意違規要 ❌）

### 🔴 動之前必問用戶
- **刪除或放寬任何鐵則**（CLAUDE.md 鐵則區、storage-and-security.md 的 🔒 規則、preflight 的 ❌ 級檢查）——就算你認為它過時了，也要先問：鐵則多半是血淚換的，文件裡看不到全部背景
- 修改升降級規則、「驗證不自驗」原則、本檔的權限分級
- 刪除 `docs/archive/` 內的任何備份
- 改動會影響正式站的檔案（script.js 等）不歸本協議管——那是日常任務，走 CLAUDE.md 常規流程

## 2. 教訓寫回哪裡、什麼格式

**踩坑後（修完 bug、被用戶糾正、subagent 連續失敗）必做**：把教訓寫進「離坑最近」的檔案末尾「教訓記錄」區：

- 專案邏輯的坑 → 對應的 `docs/project/*.md`
- 派工/模型選錯的坑 → `docs/ops/dispatch.md`
- 判斷失誤（該問沒問、該停沒停）→ `docs/ops/judgment.md`
- harness/環境的坑 → `docs/ops/diagnosis.md`

**格式（一行一條，強制）**：
```
- [YYYY-MM-DD] 症狀 → 根因 → 新規則（一句話，可執行）
```
範例：`- [2026-07-09] 搜「meta廣告」零結果 → 匯出 guard 用 if(rate&&items) 丟掉 rate=0 槽 → 兩支匯出迴圈禁用該 guard，rate 用 parseFloat 判非數字才跳過`

**升格規則**：同一條教訓若 3 個月內第二次被踩，把它從教訓區升格為該檔正文規則（🟡 級操作）；若屬「違反＝資料事故」等級，提案升格為 CLAUDE.md 鐵則（🔴 級，問用戶）。

## 3. 每次改動的必做流程

1. 改前：`cp <檔案> docs/archive/<檔名>-<YYYY-MM-DD>.bak`（僅整段改寫時；追加教訓不用）
2. 改後：檔內交叉引用的路徑跑 `ls` 驗證；改了 CLAUDE.md 或 agents 定義 → 派 verifier read-back
3. **commit + push**——短命容器，沒 push 的維護等於沒做
4. commit message 用 `docs(ops):` / `docs(project):` 前綴，一句話說清「為什麼改」

## 4. 精簡儀式（防再肥大化）

**觸發門檻**（任一成立就該精簡，🟡 級操作）：
- `CLAUDE.md` > 150 行（2026-07-11 基線 78 行）
- 任一 `docs/` 檔 > 250 行（基線最大 138 行）
- 任一「教訓記錄」區 > 15 條

**精簡動作**：重複條目合併；過時內容移到 `docs/project/history.md` 或 `docs/archive/`（**移動，不是刪除**）；教訓區把同類條目歸納成一條正文規則。
**精簡的驗收**（派 verifier）：列出精簡前後的「規則清單」（每條規則一行），逐條核對：每條舊規則要嘛還在、要嘛有移入位置、要嘛在回報中列為「提議刪除待用戶確認」。

## 5. 禁止事項

- ❌ 在 CLAUDE.md 使用 `@` import 語法展開引用檔（會把引用檔每次塞進上下文，摧毀路由設計）
- ❌ 把日常變更的 changelog 寫回 CLAUDE.md（寫 history.md）
- ❌ 「順手」在多個檔案重複同一條規則——規則只住一個家，其他地方用路徑引用；重複的副本會在下次修改時分岔成互相矛盾的版本
- ❌ 用「我覺得用不到」為由刪規則（走 🔴 流程）
- ❌ 在制度檔里寫模型行銷名或能力假設（如「這需要 Fable 級模型」）——寫可驗證的行為判準
