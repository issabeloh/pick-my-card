# 模型調度守則（主對話怎麼派工、選模型、驗收）

> 適用對象：未來每個 session 的主模型（Sonnet/Opus/Haiku 等級都適用）。
> 環境快照 2026-07-11（查證自官方文件與本環境 Agent 工具，非憑印象；環境變更時照 maintenance.md 更新本節）：
> - Agent 工具呼叫時 `model` 可選：`sonnet` / `opus` / `haiku` / `fable`（fable 未必長期可用，不可依賴）
> - **effort 無法在呼叫當下指定**，只能寫在 `.claude/agents/*.md` frontmatter 的 `effort:`（合法值 low/medium/high/xhigh/max，預設繼承主會話）。此欄位出自官方文件（code.claude.com/docs/en/sub-agents），但在本環境未實測過是否真的生效——若觀察到 agent 行為像是忽略它，當作「繼承主會話」處理並把觀察寫進本檔教訓區
> - 內建 agent：`Explore`（唯讀搜尋）、`Plan`（方案設計）、`general-purpose`、`claude-code-guide`（查 Claude Code 用法）
> - 專案自訂 agent（`.claude/agents/`）：`scout`（haiku/medium，唯讀偵察）、`builder`（sonnet/high，實作）、`verifier`（sonnet/high，fresh-context 驗收）

## 1. 指揮官不下場

主對話只做四件事：**決策、派工、整合、對用戶溝通**。以下工作一律派 subagent，主對話只收結論：

| 工作型態 | 派給 | 為什麼不自己做 |
|---|---|---|
| 廣度未知的搜尋／掃 repo／盤點呼叫端 | `scout`（或內建 `Explore`） | 搜尋過程的檔案內容會塞爆主對話，觸發壓縮後忘記鐵則 |
| 查網頁／查文件 | `general-purpose` 或 WebFetch 派工 | 網頁原文動輒數千行 |
| 批次改檔（同一模式套用到 ≥3 處） | `builder`（模式驗證後可降 `haiku`，見第 5 節） | 機械性工作不需要主對話的上下文 |
| 有明確驗收條件的實作／修 bug | `builder` | 同上＋實作過程的試錯不該污染主對話 |
| 驗收別人（或自己）宣稱完成的工作 | `verifier` | 見第 6 節「驗證不自驗」 |

**允許主對話自己動手的例外**（避免為了 10 行小事付一次冷啟動成本）：
- 已知確切位置（檔案:行號）的單點修改，預估 diff ≤30 行、涉及 ≤2 個檔案
- 讀 1–2 個**已知**檔案的**指定區段**（≤200 行）
- 跑單一指令並看結果（git、preflight、jq 一行查詢）

**判準一句話**：開工前答不出「要讀哪幾個檔案的哪些區段」→ 先派 scout；答得出且總量小 → 可以自己做。

## 2. 派工三件套（缺一不派）

每個派工 prompt 必含（模板見 `docs/ops/templates.md`，直接套）：
1. **目標與動機**：要做什麼＋為什麼（動機讓 agent 在邊角案例自己做出對的取捨）
2. **驗收條件**：可勾選的清單，每條可被機械驗證（「跑 X 看到 Y」「檔案 Z 含 W」）。寫不出驗收條件＝你自己還沒想清楚，先想清楚再派
3. **回報格式**：明確規定回什麼、多長、長產物落哪裡（見第 4 節）

另外必附：相關檔案路徑與已知行號、CLAUDE.md 路由表指到的必讀檔、已知的坑。**subagent 是冷啟動**——你不給的 context 它就沒有，別假設它「應該知道」。

## 3. 模型選擇表

| 任務 | agent / model | 理由 |
|---|---|---|
| 搜尋、定位、盤點、read-back | `scout`（haiku） | 便宜；錯了損失小、驗證容易 |
| 一般實作、重構、修 bug | `builder`（sonnet） | 預設工作馬 |
| 驗收、第二意見 | `verifier`（sonnet） | 驗收不需要比實作者聰明，需要的是 fresh context |
| 方案設計、跨模組架構取捨 | `Plan` 或 `general-purpose` 帶 `model: opus` | 需要全局推理 |
| 對抗審查、高風險判斷的第二意見 | `general-purpose` 帶 `model: opus` | 用不同（更強）模型降低共同盲點 |
| 模糊題、品味題（文案語氣、視覺取捨） | 見 `docs/ops/judgment.md` 第 6 節 | 模型堆疊補不了品味，通常該問用戶 |
| 查 Claude Code/API 用法 | `claude-code-guide` | 有官方文件工具，別憑記憶答 |

## 4. 回報合約（subagent 端已寫進定義檔，主對話負責執行）

- subagent 只回：**結論、檔案:行號、驗收逐條 ✅/❌＋證據**
- 超過 ~40 行的產物（分析報告、大段程式碼、清單）寫到 scratchpad 或 docs/，回傳**路徑**
- 主對話**不得**把 subagent 的長回報原文轉貼給用戶——整合成結論，附路徑
- subagent 回報失敗時必須帶完整失敗軌跡（做了什麼、錯誤原文）；只說「失敗了」的回報視同未回報，重派並在 prompt 裡點名要軌跡

## 5. 升降級路徑

| 情況 | 動作 |
|---|---|
| `haiku`（scout 等）出錯 **1 次**（回報錯位置、漏找、格式不遵守） | 直接升 `sonnet` 重派，不給第二次機會 |
| `sonnet`（builder 等）在**同一個子任務**連錯 **2 次** | 升 `opus` 重派，prompt 附上**完整失敗軌跡**（兩次的 prompt、diff、錯誤輸出、驗收哪條沒過）——沒有軌跡的升級會重蹈覆轍 |
| 某模式已被驗證可行（例：一種修法在 2 處手工驗證通過） | **降級**：剩餘的批次套用派 `haiku`/`sonnet`，每處附上已驗證的範例 diff 當樣板 |
| 同一件事已重試 **2 輪**（不論模型） | **停**。不是模型不夠強，是方向可能錯了——讀 `docs/ops/judgment.md` 第 4 節，換路或問用戶 |

「同一個子任務」的判定：驗收條件同一條沒過，就算同一個子任務；換了症狀但根因相同也算。

## 6. 驗證不自驗

**寫的人不能當自己的驗收者**——實作時的假設會原封不動帶進驗證。規則：

- **檔案/文件產出** → 派 `verifier` read-back：實際讀檔逐條核驗收條件（不是問實作者「寫了嗎」）
- **程式碼產出** → 實跑優先序：`tools/preflight.sh` → 相關的 `docs/ops/regression.md` 條目 → 起 server 實際操作。「程式碼看起來對」不是驗證
- **高風險判斷**（動用戶資料、動 firestore.rules、動匯出邏輯、不可逆操作）→ 加第二意見：派 `opus` 獨立分析同一問題，不給它第一個結論；兩者矛盾時呈給用戶裁決
- **多答案評審**（適用於「有多種合理解」的問題，如命名、文案、方案選型）：派 2–3 個 agent 各自獨立產出，再派一個 fresh agent 當評審選優＋說理由

主對話自己做的小改動也一樣：至少跑 preflight＋自己 diff 一遍找 debug 殘留，重要的照樣派 verifier。

## 7. 並行與成本

- 相互獨立的派工**同一批發出**（Agent 呼叫可並行），不要串行等
- 但**同一批 agent 不可寫同一個檔案**——會互相覆蓋；有寫入重疊的任務串行派
- 每次派工都有冷啟動成本（agent 要重新讀規則檔）：能合併成一個 prompt 的相鄰小任務就合併，不要為每個小問題各開一個 agent
