# 派工 Prompt 模板（複製→填空→派出）

> 配合 `docs/ops/dispatch.md` 使用：模板決定「說什麼」，dispatch 決定「派給誰」。
> `【】` 是必填空格；(選填) 可刪。填不出【驗收條件】＝任務還沒想清楚，先想清楚再派。
> 所有模板已內建回報合約——不要刪掉回報格式那段。

## 1. 搜尋／定位（派 scout；haiku）

```
目標：找出【要找什麼：函數/呼叫端/資料欄位/樣式規則】。
動機：【為什麼要找：接下來要改什麼，讓你判斷哪些結果相關】。
範圍：【檔案或目錄；不確定就寫「全 repo」】。已知線索：【關鍵字/區塊目錄詞/大約位置】(選填)。
注意：script.js 有 11,900 行，先 Grep 再讀區段；cards.data 要 base64 -d 後用 jq。

驗收條件：
- [ ] 每個結果都有 檔案:行號
- [ ] 明確回答：【要回答的具體問題，如「有幾個呼叫端？各在哪？」】
- [ ] 找不到時列出你搜過的 pattern（至少 3 種變體）

回報格式：每個發現一行「檔案:行號 — 說明」；最後一段「結論」直接回答上述問題；全文 ≤40 行，超出部分寫檔回傳路徑。
```

## 2. 實作／修 bug（派 builder；sonnet）

```
目標：【做什麼，一句話】。
動機：【為什麼：用戶回報了什麼/哪裡壞了/要支援什麼新場景】。
必讀：CLAUDE.md 鐵則＋【依路由表填：docs/project/xxx.md 第 N 節】。
相關位置：【檔案:行號 或 Grep 關鍵字】。
已知的坑：【從領域檔/教訓記錄抄相關條目】(選填)。
範圍限制：只改【檔案清單】；不要動【明確排除的東西，如「不要動 levelSettings 的讀取邏輯」】。

驗收條件：
- [ ] 【功能行為：做 X 操作看到 Y 結果——要具體到可實際操作】
- [ ] 【回歸：regression.md 第 N 條結果不變】
- [ ] bash tools/preflight.sh 通過
- [ ] 沒有 debug 殘留（自己 diff 一遍）

回報格式：結果一句話；改動清單（檔案:行號範圍—改了什麼）；驗收逐條 ✅/❌＋證據；preflight 輸出原文。失敗時附完整軌跡（試了什麼、錯誤原文），不要只說失敗。
```

## 3. 重構（派 builder；sonnet。跨 ≥3 模組的先派 Plan 出方案）

```
目標：把【現狀】重構為【目標形態】。
動機：【為什麼值得動：重複了幾份/哪次改動被它拖慢】。
不變式（重構的定義——這些必須完全不變）：
- [ ] 【對外行為 1，如「12 組回歸搜尋結果逐字一致」】
- [ ] 【對外行為 2，如「詳情頁各卡顯示不變」】
必讀：【docs/project/xxx.md】。本專案先例：mergeResultsByActivity（4 份合 1）、extractPlaceholderField（3 份正則合 1），風格照它們。
步驟要求：小步走，每步可獨立驗證；一次 commit 一個等價變換。

驗收條件：
- [ ] 上述不變式逐條驗證通過（說明驗證方式）
- [ ] 舊實作已刪乾淨：Grep 舊函數名/舊 key 零殘留
- [ ] bash tools/preflight.sh 通過

回報格式：同實作模板。若中途發現不變式保不住 → 停手回報，不要硬改。
```

## 4. 研究／查證（派 general-purpose；查 Claude Code 用法派 claude-code-guide）

```
問題：【要回答什麼，越具體越好】。
動機：【答案會決定什麼：要選方案/要寫文件/要回覆用戶】。
資訊來源優先序：【官方文件 > repo 內文件 > 程式碼現狀 > 網路文章】。
判準：查不到就明說「查無」＋查過哪裡；【需要多新的資訊：如「以 2026 年現行文件為準」】。

驗收條件：
- [ ] 每個結論都標出處（URL/檔案:行號）
- [ ] 區分「文件明載」vs「從行為推斷」vs「查無依據」三種確定度
- [ ] 回答了上述問題本身（不是相關背景的堆砌）

回報格式：先一段直接回答；再列證據（每條一行＋出處）；全文 ≤40 行，長引文寫檔回傳路徑。
```

## 5. 審查／驗收（派 verifier；sonnet。高風險加派 opus 第二意見）

```
待驗對象：【branch/commit/檔案清單】。宣稱完成的內容：【原任務的驗收條件，逐條抄過來】。
你沒看過實作過程，這是刻意的——只相信檔案現狀與實跑結果。

驗收條件（對每條給 ✅/❌＋證據，無證據不准 ✅）：
- [ ] 【逐條抄原驗收條件】
- [ ] 額外必查：是否違反 CLAUDE.md 鐵則；debug 殘留；文件內路徑引用用 ls 驗證存在
- [ ] 【文件類：read-back 核對內容自洽、規則之間不打架】
- [ ] 【程式類：跑 preflight＋regression 相關條目＋可實跑的實跑】

回報格式：第一行「結論：通過／不通過」；不通過列出項目＋證據；全文 ≤50 行。只報告，不修復。
```

## 使用備註

- 派出前自檢：這個 prompt 單獨拿給一個「完全沒有本對話上下文」的人，做得出來嗎？做不出來就是 context 給少了
- 同一批並行派工不可寫同一檔案（dispatch.md 第 7 節）
- 模板本身要改進：照 `docs/ops/maintenance.md` 的規則改（模板措辭可自行改，欄位增刪要謹慎）
